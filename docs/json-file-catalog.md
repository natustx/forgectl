# JSON File Catalog

All JSON files that forgectl consumes, produces, or manages.

## Files

### spec-queue.json

| Property | Value |
|----------|-------|
| Phase | specifying |
| Type | Input |
| Consumed by | `forgectl init --phase specifying --from <path>` |
| Validated at | `init`, `validate` |
| Top-level key | `"specs"` |
| Schema reference | [schemas/spec-queue.md](schemas/spec-queue.md) |

Defines the queue of specifications to draft. Each entry names a spec with its topic of concern, domain, target file path, planning sources, and dependency ordering.

### plan-queue.json

| Property | Value |
|----------|-------|
| Phase | generate_planning_queue (produced), planning (consumed) |
| Type | Input |
| Location | `<state_dir>/plan-queue.json` (auto-generated); or user-supplied path via `--from` override |
| Consumed by | `forgectl init --phase planning --from <path>`, `forgectl advance --from <path>` (at specifying or generate_planning_queue PHASE_SHIFT) |
| Validated at | `init`, phase shift, `validate` |
| Top-level key | `"plans"` |
| Schema reference | [schemas/plan-queue.md](schemas/plan-queue.md) |

Defines the queue of implementation plans to produce. Each entry names a plan with its domain, target file path, spec references for study, spec commit hashes, and code search roots.

Auto-generated by forgectl in the `generate_planning_queue` phase: at ORIENT, the scaffold groups completed specs by domain and writes the queue to `<state_dir>/plan-queue.json`. The architect reviews and edits the file at REFINE, then advances to transition to planning.

Alternatively, a user-supplied file can be provided via `--from` at the specifying PHASE_SHIFT (which skips the generate_planning_queue phase entirely) or at the generate_planning_queue PHASE_SHIFT (which overrides the auto-generated file).

Note: the `topic` field was removed from this schema. `spec_commits` was added to carry commit hashes from completed specs.

### plan.json

| Property | Value |
|----------|-------|
| Phase | planning (produced), implementing (consumed and mutated) |
| Type | Internal |
| Consumed by | `forgectl advance` (VALIDATE state in planning), `forgectl init --phase implementing --from <path>`, `forgectl advance` (planning→implementing phase shift) |
| Validated at | VALIDATE state (planning phase), phase shift to implementing, `init --phase implementing`, `validate` |
| Top-level key | `"context"` (with `"refs"`, `"layers"`, `"items"`) |
| Schema reference | [schemas/plan-json.md](schemas/plan-json.md) |

The implementation plan produced during the planning phase. Contains context, reference files, dependency layers, and individual work items with tests. During the transition to implementing, forgectl mutates each item by adding `passes: "pending"` and `rounds: 0`.

**Validation checks (12):**

1. JSON parse — valid JSON
2. Top-level fields — only `context`, `refs`, `layers`, `items`
3. Context fields — `domain` and `module` required, non-empty
4. Refs exist — every `refs[].path` resolves to an existing file
5. Item schema — every item has `id`, `name`, `description`, `depends_on`, `specs`, `refs`, `files`, `tests`
6. Item ID uniqueness — no duplicate IDs
7. Layer coverage — every item in exactly one layer; every layer item ID exists
8. Layer ordering — items only depend on items in equal or earlier layers
9. DAG validity — no cycles in `depends_on` graph
10. Test schema — every test has `category` and `description`; category is `functional`, `rejection`, or `edge_case`
11. Test categories — at least one test per item
12. Notes files — every path in `items[].refs` resolves to an existing notes file (relative to plan.json directory)

### forgectl-state.json

| Property | Value |
|----------|-------|
| Phase | all |
| Type | Internal (persistent session state) |
| Location | `<project_root>/.forgectl/state/forgectl-state.json` (configurable via `config.paths.state_dir`) |
| Validated at | every command (read on startup), crash recovery |
| Top-level key | `"phase"` (with `"state"`, `"config"`, phase-specific sections) |
| Schema reference | [schemas/forge-state.md](schemas/forge-state.md) |

The session state file. Written atomically (`.tmp` → rename) with `.bak` backup for crash recovery. Config is locked at init (except `config.general.user_guided`). The `phase` field determines which state names and transitions are valid. Contains `session_id` (UUID v4) at root level, generated at init.

Not a target for `forgectl validate` — the state file is managed internally and validated on every read.

## Auto-Detection

The `forgectl validate` command auto-detects file type from the top-level JSON key:

| Top-level key | Detected type | File |
|---------------|--------------|------|
| `"specs"` | spec-queue | spec-queue.json |
| `"plans"` | plan-queue | plan-queue.json |
| `"context"` | plan | plan.json |

Use `--type <type>` to override auto-detection when needed. Valid types: `spec-queue`, `plan-queue`, `plan`.

## Eval Reports

Eval reports are markdown files written by sub-agents. Forgectl records their paths but does not read or validate their content.

| Phase | Convention |
|-------|-----------|
| specifying | `<domain>/specs/.eval/batch-N-rM.md` |
| specifying (cross-ref) | `<domain>/specs/.eval/cross-reference-rN.md` |
| planning | `<domain>/.forge_workspace/implementation_plan/evals/round-N.md` |
| implementing | `<domain>/.forge_workspace/implementation_plan/evals/batch-N-round-M.md` |
