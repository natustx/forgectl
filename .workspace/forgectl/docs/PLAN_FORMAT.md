# Implementation Plan Format

> Defines the JSON structure for implementation plans generated by the planning scaffold.
> An implementation plan is a lightweight manifest that sequences work items, references specs and notes for detail, and defines acceptance criteria through structured test cases.

---

## Principles

1. **The JSON is a roadmap, not a textbook.** It sequences items and tracks progress. Heavy content (code approaches, design decisions, test procedures) lives in reference notes files.
2. **Specs are the source of truth.** Items point to spec sections for the contract. The plan does not duplicate spec content.
3. **Items are small.** Each item is one type, one function, one test file — something an agent can complete in a single pass. If an item feels too large, break it down.
4. **The plan is immutable after acceptance.** Items are not added or removed after the planning scaffold accepts the plan — that requires re-planning. The implementation scaffold adds its own tracking fields (`passes`, `rounds`) during init.

---

## File Structure

An implementation plan lives in the domain's workspace directory:

```
<domain>/.workspace/implementation_plan/
├── plan.json          # The implementation plan manifest
└── notes/             # Reference notes per package
    ├── config.md
    ├── daemon.md
    ├── launch.md
    └── ...
```

- `plan.json` is the manifest consumed by the scaffold.
- `notes/` contains implementation guidance referenced by items via the `ref` field.
- Notes files are organized by package, not by item. Multiple items may reference different sections of the same notes file.

---

## Schema

### Top Level

```json
{
  "context": { ... },
  "refs": [ ... ],
  "layers": [ ... ],
  "items": [ ... ]
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `context` | object | yes | Domain and project metadata the agent needs before reading any item |
| `refs` | string[] | yes | Paths to spec files and notes files referenced by items |
| `layers` | object[] | yes | Ordered build layers grouping items by dependency tier |
| `items` | object[] | yes | The implementation work items |

### Context

Project metadata that orients the agent. The agent reads this once at the start.

```json
{
  "context": {
    "domain": "launcher",
    "module": "spectacular/launcher",
    "go_version": "1.21.6",
    "binary": "spectacular-launcher"
  }
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `domain` | string | yes | Domain name (matches the top-level directory) |
| `module` | string | yes | Go module name (or package name for Python/TypeScript) |
| `go_version` | string | no | Go version for go.mod. Omit for non-Go domains. |
| `binary` | string | no | Binary name if the domain produces an executable |

Context fields are domain-specific. Add fields as needed for the domain's language and tooling (e.g., `python_version`, `node_version`, `framework`). Keep it minimal — only what the agent cannot derive from the codebase.

### Refs

A flat list of file paths referenced by items. Includes both spec files and notes files. Paths are relative to the project root.

```json
{
  "refs": [
    "launcher/specs/service-configuration.md",
    "launcher/specs/launching-system-processes.md",
    "launcher/specs/stopping-system-processes.md",
    "launcher/.workspace/implementation_plan/notes/config.md",
    "launcher/.workspace/implementation_plan/notes/launch.md"
  ]
}
```

This serves as an inventory — the agent can verify all referenced files exist before starting work.

### Layers

Ordered dependency tiers that define the build sequence. Each layer lists the items that belong to it. Layers are sequential — all items in a layer must be complete before items in the next layer can begin. Items within the same layer have no ordering constraint and may be implemented in parallel.

```json
{
  "layers": [
    { "id": "L0", "name": "Foundation", "items": ["bootstrap", "config.types", "config.load", "config.tests"] },
    { "id": "L1", "name": "Core", "items": ["daemon.types", "daemon.io", "daemon.tests", "spawn", "health", "launch", "launch.tests"] },
    { "id": "L2", "name": "CLI", "items": ["stop", "stop.tests", "cli.start", "cli.stop", "cli.root", "cli.tests"] }
  ]
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | string | yes | Layer identifier (e.g., "L0", "L1") |
| `name` | string | yes | Human-readable layer name |
| `items` | string[] | yes | Item IDs belonging to this layer, in suggested implementation order |

**Constraints:**
- Every item ID in the `items` array must exist in the top-level `items` array.
- Every item in the top-level `items` array must appear in exactly one layer.
- Layer ordering is significant — L0 before L1 before L2.

### Items

The individual work units. Each item is small enough for an agent to complete in a single focused pass.

```json
{
  "id": "config.load",
  "name": "Load YAML, apply defaults, validate strictly",
  "description": "Parse spectacular.yml, apply default host/port values, reject invalid config with structured errors naming the offending field.",
  "spec": "service-configuration.md#behavior-loading",
  "ref": "notes/config.md#load",
  "depends_on": ["config.types"],
  "files": [
    "internal/config/load.go",
    "internal/config/load_test.go"
  ],
  "steps": [
    "Implement LoadConfig() using goccy/go-yaml strict mode",
    "Add default port logic (portal=8080, api=8081, optimizer=8082)",
    "Add post-unmarshal validation for port range and empty host",
    "Write table-driven tests for valid, rejection, and edge cases"
  ],
  "tests": [
    { "category": "functional", "description": "Default ports applied when services are empty objects" },
    { "category": "rejection", "description": "Missing services section rejected" },
    { "category": "edge_case", "description": "Duplicate ports allowed" }
  ]
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | string | yes | Unique identifier. Convention: `<package>.<concern>` (e.g., `config.load`, `daemon.io`) |
| `name` | string | yes | Short action-oriented name |
| `description` | string | yes | One or two sentences explaining what this item does and why |
| `spec` | string or string[] | no | Spec file reference(s) with optional `#section` anchor. Relative to the spec filename listed in `refs`. Null for items with no spec (e.g., bootstrap, test scaffolding). When an item touches multiple spec sections, use an array. |
| `ref` | string | no | Notes file reference with `#section` anchor. Relative to the notes directory. Null for items where the spec is sufficient guidance. |
| `depends_on` | string[] | yes | Item IDs that must be complete before this item can begin. Empty array for items with no dependencies. |
| `files` | string[] | yes | File paths to create or modify for this item. Relative to the domain root. Empty array for items that don't touch files directly (e.g., integration test items that only run existing code). |
| `steps` | string[] | yes | Ordered implementation instructions. Action-oriented descriptions of what to build, in the sequence the engineer should follow. |
| `tests` | object[] | yes | Structured test cases that define acceptance criteria for the item. |

#### Item ID Conventions

- Use dot notation: `<package>.<concern>` (e.g., `config.types`, `config.load`, `config.tests`)
- Bootstrap/scaffolding items use a plain name: `bootstrap`
- Test items use `.tests` suffix: `config.tests`, `launch.tests`
- Keep IDs short and scannable

#### Spec References

The `spec` field points to spec sections the agent should read for the contract. Use the spec filename (not the full path) with an optional `#section` anchor:

```json
"spec": "service-configuration.md#behavior-loading"
```

For items spanning multiple sections:

```json
"spec": [
  "launching-system-processes.md#startup-sequence",
  "launching-system-processes.md#spawning-a-detached-process",
  "launching-system-processes.md#health-checking-a-process"
]
```

The full path is resolved from the `refs` list.

#### Notes References

The `ref` field points to a section in a notes file for implementation guidance:

```json
"ref": "notes/config.md#load"
```

Notes files are markdown with `##` headings used as anchors. The section contains code approaches, design decisions, library patterns, and test strategies that are too verbose for the JSON.

### Tests

Structured test cases that define acceptance criteria for an item. Tests are descriptive — they tell the engineer and evaluator what must be true, not whether it is true yet.

```json
{
  "category": "functional",
  "description": "Default ports applied when services are empty objects"
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `category` | string | yes | One of: `functional`, `rejection`, `edge_case` |
| `description` | string | yes | What this test verifies. Should be specific enough to evaluate unambiguously. |

#### Test Categories

Categories map to spec structure:

| Category | Maps to | Purpose |
|----------|---------|---------|
| `functional` | Spec §Behavior sections | Core happy-path behavior works correctly |
| `rejection` | Spec §Rejection table | Invalid inputs are rejected with correct error signals |
| `edge_case` | Spec §Edge Cases | Boundary conditions and unusual scenarios are handled |

#### Test Descriptions

Write descriptions as assertions — what must be true, not what to do:

- Good: `"Default ports applied when services are empty objects"`
- Good: `"Missing services section rejected"`
- Bad: `"Test the config loading"` (too vague)
- Bad: `"Create a YAML file and load it"` (describes steps, not the assertion)

Detailed verification procedures (step-by-step instructions for how to verify) belong in the notes file, not in the test description.

#### Steps

Steps are ordered implementation instructions — action-oriented descriptions of what to build. They tell the engineer *what to do*, complementing tests which tell the engineer *what must be true when done*.

Write steps as directives:

- Good: `"Implement LoadConfig() using goccy/go-yaml strict mode"`
- Good: `"Add post-unmarshal validation for port range and empty host"`
- Bad: `"The config loader"` (not actionable)
- Bad: `"Default ports applied when services are empty objects"` (that's a test, not a step)

Detailed code approaches and design decisions belong in the notes file. Steps should be concise enough to scan but specific enough to act on.

#### Files

The `files` field lists file paths the engineer will create or modify for this item. Paths are relative to the domain root.

```json
"files": [
  "internal/config/load.go",
  "internal/config/load_test.go"
]
```

This gives the engineer and evaluator a concrete scope for the item. Use an empty array for items that don't directly touch files (e.g., integration test items that exercise existing code).

## Notes Files

Notes files are markdown documents organized by package. They provide implementation guidance that is too verbose for the JSON manifest.

### Structure

```markdown
# Config Notes

## types
Three named struct fields, not a map. Invariant 1 (exactly three services)
is a compile-time guarantee.

## load
Use goccy/go-yaml strict mode (DisallowUnknownField) for unknown key
rejection. Post-unmarshal validation for port range and empty host.

Default ports: portal=8080, api=8081, optimizer=8082.

## test-table
Table-driven subtests. Each row: name, YAML input, expected result or error substring.
| Name | Input | Expect |
|------|-------|--------|
| valid complete | custom host:port for all 3 | values match |
| default ports | empty objects | 8080, 8081, 8082 |
| missing services | `other: true` | error contains "services" |
```

### Conventions

- One notes file per package (e.g., `notes/config.md` for `internal/config/`).
- Use `##` headings as anchors. Items reference them via `ref: "notes/config.md#load"`.
- Include code approaches when the "how" isn't obvious (e.g., `O_CREATE|O_EXCL` for atomic file creation, `SysProcAttr.Setpgid` for detached processes).
- Include test tables when the item has many test cases.
- Keep notes focused on engineering decisions, not spec content. Don't repeat what the spec says — explain how to implement what the spec requires.

## Example

A minimal plan with two items:

```json
{
  "context": {
    "domain": "launcher",
    "module": "spectacular/launcher",
    "go_version": "1.21.6"
  },
  "refs": [
    "launcher/specs/service-configuration.md",
    "launcher/.workspace/implementation_plan/notes/config.md"
  ],
  "layers": [
    { "id": "L0", "name": "Foundation", "items": ["config.types", "config.load"] }
  ],
  "items": [
    {
      "id": "config.types",
      "name": "ServiceEndpoint and ServicesConfig structs",
      "description": "Go structs for validated service endpoint configuration. Three named fields enforce the three-services invariant at compile time.",
      "spec": "service-configuration.md#interface-outputs",
      "ref": "notes/config.md#types",
      "depends_on": [],
      "files": [
        "internal/config/types.go"
      ],
      "steps": [
        "Define ServiceEndpoint struct with Host (string) and Port (int) fields",
        "Define ServicesConfig struct with three named ServiceEndpoint fields: Portal, API, Optimizer",
        "Add YAML struct tags for deserialization"
      ],
      "tests": [
        { "category": "functional", "description": "Three named fields, not a map" }
      ]
    },
    {
      "id": "config.load",
      "name": "Load YAML, apply defaults, validate strictly",
      "description": "Parse spectacular.yml, apply default host/port values, reject invalid config with structured errors naming the offending field.",
      "spec": "service-configuration.md#behavior-loading",
      "ref": "notes/config.md#load",
      "depends_on": ["config.types"],
      "files": [
        "internal/config/load.go",
        "internal/config/load_test.go"
      ],
      "steps": [
        "Implement LoadConfig() using goccy/go-yaml strict mode",
        "Add default port logic (portal=8080, api=8081, optimizer=8082)",
        "Add post-unmarshal validation for port range and empty host",
        "Write table-driven tests for valid, rejection, and edge cases"
      ],
      "tests": [
        { "category": "functional", "description": "Default ports applied when services are empty objects" },
        { "category": "functional", "description": "Default host applied when only port specified" },
        { "category": "rejection", "description": "Missing services section rejected" },
        { "category": "rejection", "description": "Port out of range rejected" },
        { "category": "rejection", "description": "Unknown keys rejected" },
        { "category": "edge_case", "description": "Empty file rejected" },
        { "category": "edge_case", "description": "Duplicate ports allowed" }
      ]
    }
  ]
}
```
